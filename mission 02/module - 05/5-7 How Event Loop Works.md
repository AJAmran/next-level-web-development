JavaScript single-threaded হওয়া সত্ত্বেও কীভাবে একই সাথে অনেকগুলো কাজ (যেমন: API call, Timer, User Input) সামলায়, তার আসল রহস্য হলো **Event Loop**।

সহজ কথায় বলতে গেলে, Event Loop হলো একজন ট্রাফিক পুলিশের মতো, যে ঠিক করে কখন কোন গাড়ি (Task) রাস্তা (Call Stack) দিয়ে যাবে।

নিচে বিস্তারিত এবং "ভেঙে ভেঙে" আলোচনা করা হলো:

---

## ১. প্রধান ৪টি অংশ (The Core Components)

Event Loop বুঝতে হলে আগে এই চারটি জিনিস বুঝতে হবে:

* **Call Stack:** এটা একটা স্ট্যাক (Stack) যেখানে আপনার কোডগুলো রান হয়। এখানে **LIFO (Last In, First Out)** নিয়ম চলে। অর্থাৎ, যে ফাংশন সবার শেষে ঢোকে, সে সবার আগে বের হয়।
* **Web APIs:** ব্রাউজারের দেওয়া কিছু স্পেশাল ফিচার (যেমন: `setTimeout`, `fetch`, DOM Events)। এগুলো JavaScript ইঞ্জিনের বাইরে থাকে।
* **Callback Queue (Task Queue):** যখন কোনো Web API-এর কাজ শেষ হয়, তখন তার রেজাল্ট বা callback ফাংশনটি এই লাইনে এসে দাঁড়ায়।
* **Microtask Queue:** এটি একটি স্পেশাল লাইন (VIP Line)। এখানে সাধারণত **Promises** এবং `MutationObserver`-এর কাজগুলো জমা হয়।

---

## ২. এটি কীভাবে কাজ করে? (Step-by-Step Flow)

পুরো প্রক্রিয়াটি নিচের ধাপগুলোতে ঘটে:

### ধাপ ১: Call Stack-এ কাজ শুরু

যখন আপনি কোনো কোড রান করেন, সেটি **Call Stack**-এ জমা হয় এবং সাথে সাথে এক্সিকিউট হওয়া শুরু করে।

### ধাপ ২: Async কাজ শনাক্ত করা

যদি স্ট্যাকে এমন কোনো কাজ আসে যা সময় নেবে (যেমন: `setTimeout` বা API call), তবে JavaScript সেটা নিজে না করে **Web APIs**-এর হাতে ছেড়ে দেয়। ব্রাউজার তখন ব্যাকগ্রাউন্ডে সেই কাজটা করতে থাকে।

### ধাপ ৩: Queue-তে জমা হওয়া

ব্যাকগ্রাউন্ডে কাজ শেষ হলে, Web API সেই ফাংশনটিকে সরাসরি স্ট্যাকে পাঠাতে পারে না। সেটিকে পাঠিয়ে দেয় **Callback Queue**-তে।

### ধাপ ৪: Event Loop-এর নজরদারি

এইখানেই **Event Loop**-এর এন্ট্রি। সে সারাক্ষণ দুটি জিনিস চেক করে:

1. **Call Stack** কি একদম ফাঁকা?
2. **Queue**-তে কি কোনো কাজ অপেক্ষা করছে?

### ধাপ ৫: এক্সিকিউশন

যদি Call Stack ফাঁকা থাকে, তবে Event Loop সবার আগে **Microtask Queue** (Promises) থেকে কাজগুলো স্ট্যাকে পাঠায়। সেগুলো শেষ হলে সে **Callback Queue** থেকে কাজগুলো একে একে স্ট্যাকে তুলে দেয়।

## ৩. একটি বাস্তব উদাহরণ (Code Example)

নিচের কোডটি দেখলে কনফিউশন একদম ক্লিয়ার হয়ে যাবে:

```javascript
console.log("১. শুরু");

setTimeout(() => {
  console.log("২. টাইমার শেষ (Macro-task)");
}, 0);

Promise.resolve().then(() => {
  console.log("৩. প্রমিজ সলভড (Micro-task)");
});

console.log("৪. শেষ");

```

### আউটপুট কেন এমন হবে?

| ক্রম | কি ঘটছে? | ব্যাখ্যা |
| --- | --- | --- |
| **১. শুরু** | Synchronous | সরাসরি Call Stack-এ গিয়ে প্রিন্ট হয়ে গেল। |
| **৪. শেষ** | Synchronous | `setTimeout` এবং `Promise` কে ব্যাকগ্রাউন্ডে পাঠিয়ে পরের লাইনে চলে গেল এবং প্রিন্ট হলো। |
| **৩. প্রমিজ** | Microtask | Promise-এর প্রায়োরিটি বেশি, তাই Call Stack ফাঁকা হতেই এটি আগে রান হলো। |
| **২. টাইমার** | Callback Task | যদিও সময় `0` ছিল, তাও সে লাইনের শেষে (Task Queue) ছিল, তাই সবার শেষে রান হলো। |

| অংশ | কাজ | সহজ উদাহরণ |
| --- | --- | --- |
| **Call Stack** | বর্তমান কাজ করা | রান্নার চুলা (একবার এক পাতিল) |
| **Web API** | বাইরের সাহায্য নেওয়া | ওভেন বা ফ্রিজ |
